## 1 实验目的

通过实测与分析，理解以下几种 UNIX/Linux I/O 方式的性能差异及其背后的系统原理：

1. 系统调用 `read()` 在不同 BUFFSIZE 下的读取效率；
2. 标准 I/O 库逐字符函数 `getc()` / `fgetc()` 的读取效率，以及宏实现与函数实现的差异；
3. 标准库块读取函数 `fread()` 在不同 BUFFSIZE 下的读取效率；
4. 自行实现的带缓冲读取函数 `my_fread()` 的效率表现；
5. `write()` 在有无 `O_SYNC` 标志下的写入效率差异。

核心目标是把"实测数据"与"系统调用开销、内核页缓存、用户态缓冲"等原理对应起来，
解释**为什么**会出现这样的性能曲线。

## 3 实验设计与方法

### 3.1 总体设计与程序流程

测试程序为单文件 C 程序 `partA/main.c`，整体流程分为三个阶段：

```
① 确定版本环境  →  ② 分别调用各测试函数  →  ③ 聚合并输出结果
```

- **① 确定版本环境**：`detect_environment()` 通过 `uname()`、`statfs()`、
  `/sys/dev/block/*/queue/rotational`、`sysconf()` 等接口收集并打印操作系统、
  内核版本、文件系统类型、磁盘类型等信息（即第 2 节内容）。
- **② 分别调用测试函数**：对 6 个测试项分别实现独立的测试函数，按 BUFFSIZE
  循环逐一测量。
- **③ 聚合并输出结果**：每完成一次测量即打印一行对齐表格，并把结果汇总写入
  CSV 文件 `results.csv`，便于后续绘图。

### 3.2 计时方法

每个测试函数在被测 I/O 循环的**前后**各取一次"快照"，二者相减得到三种时间：

| 指标 | 含义 | 采集方式 |
|------|------|----------|
| real（实时/墙钟时间） | 测量区间实际流逝的时间 | `clock_gettime(CLOCK_MONOTONIC)` |
| user（用户态 CPU 时间） | 进程在用户态消耗的 CPU 时间 | `getrusage(RUSAGE_SELF)` 的 `ru_utime` |
| sys（内核态 CPU 时间） | 进程执行系统调用、在内核态消耗的 CPU 时间 | `getrusage(RUSAGE_SELF)` 的 `ru_stime` |

三者关系大致为 `real ≈ user + sys + 阻塞等待时间`。其中 `sys` 直接反映系统调用开销，
是分析 `read()`/`write()` 性能的关键指标；选用 `CLOCK_MONOTONIC` 是为了避免系统
时钟被调整时影响测量。

### 3.3 BUFFSIZE 的选取

按实验要求，对需要扫描缓冲区大小的测试项使用以下 18 种 BUFFSIZE（单位：字节）：

```
1, 2, 4, 8, 16, 32, 64, 128, 256, 512,
1024, 2048, 4096, 8192, 16384, 32768, 65536, 16777216
```

取值大体按 2 的幂递增，覆盖从"逐字节"到"远大于文件系统块（4096 字节）"再到
"16 MB 大块"的范围，便于观察性能随缓冲区增大而变化的完整趋势。

### 3.4 各测试项的实现方法

#### 3.4.1 read()（对应 2.2.1）

以 `open(path, O_RDONLY)` 打开测试文件，在计时区间内循环调用
`read(fd, buf, buffsize)` 直到返回 0（EOF），累计读取字节数。对 18 种 BUFFSIZE
各完整读取一遍文件。

#### 3.4.2 getc() / fgetc()（对应 2.2.2）

以 `fopen()` 打开文件，循环调用 `getc(fp)`（宏实现）或 `fgetc(fp)`（函数实现）
逐字符读取直到 `EOF`。两者分别测试，用于对比"宏 vs 函数"的调用开销差异。
此项不涉及 BUFFSIZE（stdio 的内部缓冲区由库自行管理）。

#### 3.4.3 fread()（对应 2.2.3）

以 `fopen()` 打开文件，循环调用 `fread(buf, 1, buffsize, fp)` 直到读完。
此处的 BUFFSIZE 是**每次 `fread` 请求的字节数**；FILE 对象内部仍有一层由 stdio
库维护的缓冲区。对 18 种 BUFFSIZE 各测一遍。

#### 3.4.4 my_fread() 的设计与实现（对应 2.2.4）

`my_fread()` 仿照标准库 `fread()` 的工作原理自行实现，**内部基于 `read()` 系统调用
维护一个用户态缓冲区**。其"文件对象"定义为：

```c
typedef struct {
    int            fd;        /* 底层文件描述符        */
    unsigned char *buf;       /* 用户态缓冲区          */
    size_t         bufsize;   /* 缓冲区容量（固定）    */
    size_t         pos;       /* 下一个待读字节的下标  */
    size_t         len;       /* 缓冲区中有效字节数    */
    int            eof;       /* 是否已到文件末尾      */
} MYFILE;
```

`my_fread()` 的核心逻辑：

1. 优先从内部缓冲区 `buf` 中拷贝数据给调用者；
2. 仅当缓冲区耗尽（`pos == len`）时，才调用一次 `read(fd, buf, bufsize)`
   向缓冲区补充数据；
3. 重复上述过程直到满足请求字节数或到达文件末尾，返回成功读取的完整成员数。

这样，多次小请求只会触发少量 `read()` 系统调用，从而把昂贵的系统调用开销摊薄——
这正是标准库 `fread()` 高效的原因。

**测试方式**：内部缓冲区容量固定为 `BUFSIZ`（8192 字节），与 stdio 默认缓冲区一致；
扫描的 18 种 BUFFSIZE 作为**每次 `my_fread` 请求的字节数**。这样与 `fread()` 测试
形成对等对比（二者内部缓冲均固定，仅请求大小变化）。

#### 3.4.5 write()（无 O_SYNC，对应 2.2.5）

以 `open(path, O_WRONLY|O_CREAT|O_TRUNC)` 打开文件，循环调用
`write(fd, buf, buffsize)` 直到写满指定总量（默认 256 MB）。此时数据写入内核
页缓存即返回，不强制落盘。对 18 种 BUFFSIZE 各测一遍。

#### 3.4.6 write()（有 O_SYNC，对应 2.2.6）

打开文件时额外加上 `O_SYNC` 标志，其余与 3.4.5 相同。`O_SYNC` 语义为：
**每次 `write()` 调用都必须等数据真正落到存储设备后才返回**。

**实现上的特殊处理**：在 `O_SYNC` 且 BUFFSIZE 很小时，写满几百 MB 需要上千万次
同步落盘，耗时不可接受。因此 `O_SYNC` 测试为每个 BUFFSIZE 限制最多
`OSYNC_MAX_WRITES = 4096` 次 `write()` 调用，写入总量上限 16 MB；实际写入字节数
随 BUFFSIZE 变化，记录在结果表的吞吐量与 CSV 的 `bytes` 列中。这样既能观察到
`O_SYNC` 随 BUFFSIZE 变化的趋势，又把测试时间控制在合理范围内。

### 3.5 实验方法学（减小误差的措施）

- **测试文件内容**：用伪随机数据填充，避免文件系统/虚拟磁盘对全零数据做稀疏或
  压缩优化而影响结果。
- **清除缓存**：读测试受内核页缓存影响极大。建议两种做法之一：
  - 测试前手动执行 `sync && echo 3 | sudo tee /proc/sys/vm/drop_caches`；
  - 运行程序时加 `-d` 选项，让程序在每次读测试前用 `posix_fadvise(POSIX_FADV_DONTNEED)`
    逐出该文件的页缓存。
  报告中需注明本次数据是"冷缓存"还是"热缓存"条件下采集的。
- **多次取平均**：程序对每个测试项只测一次；建议运行程序 3 次以上，对结果取平均
  后再填入下表，以减小偶发抖动的影响。

### 3.6 编译与运行

```bash
cd partA
make                       # 生成可执行文件 ioperf

# 正式运行（默认 512MB 读文件，约需数分钟）
sync && echo 3 | sudo tee /proc/sys/vm/drop_caches
./ioperf                   # 结果输出到终端，并汇总写入 results.csv

# 常用选项
./ioperf -h                # 查看全部选项
./ioperf -s 256 -d         # 读文件 256MB，每次读测试前逐出页缓存
make clean                 # 清理可执行文件与测试产物
```

运行结束后，`results.csv` 各列为
`test, buffsize, real, user, sys, bytes, throughput_MBps`，可直接导入
Excel / Python(matplotlib) 等工具绘制性能曲线。

### 5.1 read() 随 BUFFSIZE 的变化趋势

## 6 思考与讨论

> 本节为深入分析，待结合实测数据撰写；以下列出需回答的问题与分析方向。

**Q1：stdio 内部维护了用户态缓冲区，为什么逐字符调用 `getc()` 的性能仍远优于
逐字节调用 `read()`？**
分析方向：系统调用涉及用户态/内核态切换（上下文保存、特权级切换、可能的 TLB/缓存
影响）；`getc()` 大多数情况只是从用户态缓冲区取一个字节，仅在缓冲区耗尽时才触发
一次 `read()`，因此把系统调用开销摊薄了成千上万倍。

**Q2：`getc()` 宏实现与 `fgetc()` 函数实现之间是否存在性能差异？**
分析方向：`getc()` 通常以宏实现，可避免一次函数调用开销；`fgetc()` 是真正的函数。
结合实测的 user 时间差异讨论二者差距是否明显、为什么（编译器优化、调用开销占比等）。

**Q3：`fread()` 与 `read()`、`getc()` 相比，stdio 缓冲机制起到了什么作用？**
分析方向：`fread()` 同时受益于"批量请求"和"内部缓冲"；对比三者在小 BUFFSIZE 下的
表现，说明用户态缓冲如何减少系统调用次数。

**Q4：`my_fread()` 与 `fread()` 的对比说明了什么？**
分析方向：通过自己实现验证"缓冲区把系统调用次数降到 文件大小 / 缓冲区大小"这一
本质；讨论实现是否达到了与标准库相近的效率。

**Q5：有无 `O_SYNC` 的性能差异说明了内核缓冲区（页缓存）的什么作用？**
分析方向：无 `O_SYNC` 时 `write()` 只把数据拷入内核页缓存即返回，由内核异步回写，
因此极快；`O_SYNC` 强制每次写都落盘，性能由存储设备的同步延迟决定。讨论延迟写
带来的性能收益与数据可靠性之间的权衡。

**Q6：综合讨论——系统调用开销与缓冲的意义。**
分析方向：从本实验所有数据出发，总结"减少系统调用次数 + 合理的缓冲区大小"为何是
I/O 性能优化的核心，并指出缓冲区并非越大越好（结合 16 MB 大块的结果）。
