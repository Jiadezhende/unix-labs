# Lab 1 — Part B：高吞吐量 I/O 服务器

围绕同一个 TCP echo 工作负载，实现并对比不同并发 I/O 模型的服务器，并在纯 echo 之外引入可配置的“业务处理成本”，考察各模型在真实处理开销下的可扩展性。

## 目录结构

```
partB/
├── src/
│   ├── common.h        公共工具声明
│   ├── common.c        socket 创建 / 非阻塞 / 写满 / 日志 / 忽略 SIGPIPE / 慢任务模拟
│   ├── server_poll.c   基于 poll() 的多路复用 echo 服务器
│   ├── server_pool.c   多线程并发 echo 服务器（thread-per-connection）
│   ├── server_epoll.c  基于 epoll 的多路复用 echo 服务器（加分项）
│   └── client.c        多线程并发压测客户端
├── scripts/
│   ├── bench.sh        参数扫描压测引擎（成本 × 连接 × 并发度，可重复）
│   └── experiments.sh  几套独立实验的编排（各自输出 results_exp*.csv）
├── Makefile
└── README.md
```

```sh
make           # 编译全部到 bin/
make clean     # 清理
```

# 第 1 层　程序：服务器与客户端

## 1.1 实现了哪些服务器

三种服务器都实现同一个 TCP echo 协议（收到什么原样回发），区别只在并发 I/O 模型，从而把对比聚焦在模型本身：

| 可执行文件 | 并发模型 | 关键机制 |
|---|---|---|
| `bin/server_poll`  | 单线程 I/O 多路复用 | `poll()` 同时监视所有 fd，每轮线性扫描 |
| `bin/server_epoll` | 单线程 Reactor | `epoll` LT 模式，内核维护就绪队列，只返回就绪事件 |
| `bin/server_pool`  | 多线程并发（每连接一线程） | 每条连接独占一个线程做阻塞式 read/write |

```sh
./bin/server_poll  -p 9090
./bin/server_epoll -p 9090
./bin/server_pool  -p 9090 -m 4096

printf 'hello\n' | nc 127.0.0.1 9090   # 连通性自测：应原样回显 hello
```

## 1.2 服务器可控参数

| 参数 | 含义 | 适用 | 默认值 |
|---|---|---|---|
| `-p` | 监听端口 | 全部 | `9090` |
| `-m` | 最大并发连接数 | 仅 `server_pool` | `4096` |
| `-w` | **每请求 CPU 忙计算单位**（占用 CPU，不让出执行流） | 全部 | `0` |
| `-u` | **每请求阻塞 `usleep` 微秒**（模拟阻塞式慢 I/O） | 全部 | `0` |

`-w` / `-u` 即“业务成本旋钮”：两者可叠加，均为 `0`（默认）时退化为纯 echo。它把对比从“谁的 echo 路径短”扩展到“存在处理开销时谁更可扩展”：

- 单线程事件循环（`poll` / `epoll`）一旦在某请求上做慢任务，**整个 reactor 被卡住**，期间其它就绪连接得不到处理；
- 每连接一线程的 `server_pool` 仍能并行推进多条连接——阻塞型慢任务靠**线程并行**（睡眠不占核）、CPU 型慢任务靠**核数并行**，代价是高连接数下线程资源 / 调度压力上升。

这正是后续引入“事件循环 + 工作线程池”混合架构的动机。

```sh
./bin/server_epoll -p 9090 -u 1000   # 每请求阻塞 1ms：单线程吞吐塌陷
./bin/server_pool  -p 9090 -w 500    # 每请求 500 单位 CPU 计算
```

## 1.3 客户端可控参数

闭环负载模型：每条连接同一时刻只有一个未完成请求，发一条消息 → 收完回显 → 再发下一条，故测得的延迟即真实 RTT。

| 参数 | 含义 | 默认值 |
|---|---|---|
| `-H` | 服务器主机 | `127.0.0.1` |
| `-p` | 服务器端口 | `9090` |
| `-c` | 并发连接数（同时打开的 socket 数） | `100` |
| `-t` | 客户端压测线程数（驱动连接的线程） | `4` |
| `-n` | 总请求数（`<=0` 表示不限，仅按 `-d` 停止） | `100000` |
| `-s` | 单条消息字节数 | `64` |
| `-d` | 持续秒数（`>0` 时启用时间上限） | `0` |

输出成功请求数、错误数、吞吐量（req/s、MB/s）与延迟（avg / P50 / P95 / P99），并附一行 `CSV,...` 供脚本采集。

```sh
./bin/client -p 9090 -c 500 -t 8 -n 200000 -s 64
```

### `-t` 与 `-c` 是两个独立的压力轴（重要）

客户端把 `-c` 条连接平均分给 `-t` 个线程，每个线程在它名下的连接上**闭环收发**：发一条 → 收完回显 → 再发下一条，同一时刻只推进一条连接。关键推论：

- 一个线程任一时刻**最多 1 个在飞请求**，所以**同时压在服务器上、正在被处理的请求数 ≈ `-t`**，与 `-c` 无关。
- 当 `-c > -t` 时，多出来的连接只是**建立后空闲挂着**，几乎不产生请求，只消耗服务器维持连接的资源。

因此两个参数压的是不同的东西：

| 轴 | 压什么 | 大致等于 | 对各 server 的影响 |
|---|---|---|---|
| `-t` THREADS | **并行处理压力**：同时在飞的请求数 | 服务器需并行处理的请求数 | `pool` 能用上几个 worker 取决于它；`poll`/`epoll` 单线程，与它无关 |
| `-c` CONNS | **连接持有成本**：同时打开的 socket 数 | 服务器需维持的连接数 | `poll` 每轮 O(N) 扫描随它涨；`pool` 的线程数随它涨 |

用法对应：

- 想看**并行处理能力** → 固定 `-c`，扫 `-t`。
- 想看**海量连接的持有开销** → 把 `-t` 固定在适中值，扫 `-c`。
- 无论测哪一项，`-t` 都要足够大，否则在飞请求太少，瓶颈会落在客户端而非服务器。

# 第 2 层　引擎：`bench.sh`

在程序层之上做参数扫描：沿 **业务成本 × 连接数 × 并发度** 三类轴遍历所有组合，每个组合重复 `REPEAT` 次，把每一次运行作为一行（含 `run` 序号）写入结果 CSV，并在末尾打印按组合取均值的汇总表。

```sh
bash scripts/bench.sh                                  # 默认基线（重复 3 次）
# 阻塞型慢任务：扫 sleep 幅度、固定连接、用时间上限
SLEEP_LIST="0 200 500 1000" CONNS_LIST=64 DURATION=10 bash scripts/bench.sh
# 并发度扫描：固定慢任务，扫 THREADS
SLEEP_LIST=500 THREADS_LIST="1 2 4 8 16" CONNS_LIST=64 DURATION=10 bash scripts/bench.sh
```

## 2.1 可控参数（环境变量）

| 变量 | 控制的轴 / 维度 | 默认值 |
|---|---|---|
| `WORK_LIST` | 业务成本：CPU 忙计算单位列表 | `"0"` |
| `SLEEP_LIST` | 业务成本：阻塞 `usleep` 微秒列表 | `"0"` |
| `CONNS_LIST` | 持有成本：连接数列表 | `"100 500 1000 2000"` |
| `THREADS_LIST` | 并发度：客户端线程数列表 | `"8"` |
| `REPEAT` | 每个组合重复运行次数 | `3` |
| `DURATION` | 每次运行持续秒数（`>0` 时改用时间上限，慢任务必备） | `0` |
| `REQS` | 每次运行总请求数（`DURATION>0` 时忽略） | `200000` |
| `MSGSIZE` | 单条消息字节数 | `64` |
| `POOL_MAX` | `server_pool` 的连接上限（`-m`，高连接数实验需调大） | `4096` |
| `SERVERS` | 参与对比的服务器列表 | 三种全测 |
| `OUT` | 结果 CSV 路径 | `results.csv` |
| `PORT` | 监听端口 | `9090` |

## 2.2 输出

结果 CSV 各列（顺序即文件表头顺序）：

| 列 | 含义 |
|---|---|
| `server` | 服务器类型（`poll` / `epoll` / `pool`） |
| `work` | 每请求 CPU 忙计算单位（`-w`） |
| `sleep_us` | 每请求阻塞 `usleep` 微秒（`-u`） |
| `run` | 该组合的重复运行序号 |
| `conns` | 并发连接数（`-c`） |
| `threads` | 客户端压测线程数（`-t`） |
| `msgsize` | 单条消息字节数（`-s`） |
| `requests` | 成功请求数 |
| `errors` | 错误数 |
| `elapsed_s` | 实测耗时（秒） |
| `throughput_reqs` | 吞吐量（req/s） |
| `mbps` | 吞吐量（MB/s） |
| `avg_ms` | 平均延迟（毫秒） |
| `p50_ms` / `p95_ms` / `p99_ms` | 延迟分位数（毫秒） |

> 慢任务（`WORK`/`SLEEP` > 0）下单线程吞吐很低，固定 `REQS` 会让单格耗时极长——务必用 `DURATION` 改为时间上限。每次运行会**覆盖** `OUT` 指向的文件，不同实验请用不同 `OUT`（实验层已自动处理）。

# 第 3 层　实验：`experiments.sh`

在引擎之上，把几套**目的明确、互相独立**的情景固化下来，各自用预设的参数组合调用 `bench.sh`，输出到独立的 `results_exp*.csv`（互不覆盖）。

```sh
bash scripts/experiments.sh        # 跑全部
bash scripts/experiments.sh 2 4    # 只跑实验 2 和 4
```

## 3.1 设计的情景

| 编号 | 情景 | 扫描的轴 | 验证的核心问题 | 输出 |
|---|---|---|---|---|
| 1 | 纯 echo 基线 | 连接数 | 无业务成本下连接数可扩展性（复现基础对比） | `results_exp1_baseline.csv` |
| 2 | 阻塞型慢任务 | sleep 幅度 | 单线程被 `usleep` 串行化；`pool` 多线程“同时睡眠”（并行度受 THREADS 限制，不占核） | `results_exp2_blocking.csv` |
| 3 | CPU 型慢任务 | 计算量 | CPU 密集下 `pool` 受核数限制；与实验 2 对比“线程并行 vs 核数并行” | `results_exp3_cpu.csv` |
| 4 | 并发度扫描 | THREADS | 验证“并发在飞 ≈ THREADS”：`pool` 随 `-t` 上升、单线程持平 | `results_exp4_concurrency.csv` |
| 5 | 高连接数线程压力 | 连接数（上探 16000） | 海量空闲连接下 `pool` 的每连接线程开销拐点；`epoll` 较平 | `results_exp5_thread_pressure.csv` |

实验 1 / 5 用固定请求数，实验 2 / 3 / 4 用时间上限（`DURATION`），均默认每组合重复 3 次。各情景的具体参数见脚本内每个实验上方的注释。

> 实验 5 连接数较高，运行前建议 `ulimit -n 65535`。绝对性能依环境而定，分析重在横向对比与变化趋势。
